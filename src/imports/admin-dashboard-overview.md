너는 제품 디자이너(UX/UI) + 운영OS 설계자 + 접근성 전문가 + 데이터/테이블 UX 전문가다.
이전 대화에서 설계한 “고객(조합원)용 웹앱”의 디자인 시스템(3색 팔레트, 이미지 우선, 일반/어르신 모드, 웹/모바일 반응형, 공지/커뮤니티/문의 흐름)을 참고하여,
이번에는 동일한 서비스와 1:1로 연동되는 “관리자(Admin) 웹앱”을 설계/생성하라.

관리자 웹앱의 목표:
- 조합 운영자가 “조합원/입금/서비스/공지&커뮤니티/문의”를 하루 운영량(예: 서비스 신청 500건/일)에서도 안전하고 빠르게 처리할 수 있게 한다.
- 데이터 탐색(검색/필터/저장뷰), 대량 처리(벌크), 작업 흐름(배정/상태/캘린더), 감사(로그/히스토리), 실수 방지(소프트 삭제/복구/권한)를 최우선으로 한다.
- 고객용 웹앱과 동일한 데이터 구조(조합원 상태, 서비스 신청 상태, 게시물 공개 상태)가 유지되어 “관리자에서 바꾸면 고객 화면에 즉시 반영되는 느낌”을 만든다.
- 관리자는 주로 Desktop에서 사용하지만, Mobile에서도 “긴급 처리(오늘 일정/문의 답변/조합원 검색/전화)”는 가능해야 한다.

────────────────────────────────────────
0) 전제 / 범위
────────────────────────────────────────
- 결과물은 “관리자 전용 웹앱”이다(고객용 웹앱과 별도 진입 URL/경로).
- 고객용 웹앱의 브랜드 톤(신뢰/편안/명확)을 유지하되, Admin은 정보 밀도가 높으므로 ‘중립색 중심 + 구조적 레이아웃’으로 운영 UI를 설계한다.
- 관리자에서는 어르신 모드가 필수는 아니지만, 실제 운영자가 고령일 수 있으므로 “관리자 큰 글자 모드(Operator Large Mode)”를 옵션으로 제공한다.
  - 단, 고객용 “어르신 모드”와 혼동되지 않게 이름을 다르게 한다.
  - 고객용 콘텐츠 편집 시에는 “일반용 문구/어르신용 문구(요약)” 필드를 편집할 수 있어야 한다.

  - 주의/대기: Accent 배지 + 느낌표 아이콘

[타이포]

- 테이블 가독성: 행 높이(기본 44px, 큰글자 52px), 줄임표 처리 + hover tooltip

[컴포넌트 스타일]
- 버튼:
  - Primary(채움): Primary 배경 + 흰 글자
  - Secondary(테두리): Primary 테두리 + Primary 글자
  - Tertiary(텍스트): Primary 글자
  - Destructive는 빨강 금지 → “경고 아이콘 + ‘삭제’ 텍스트 + 2단계 확인 모달”로 처리
- 배지(Chip/Badge):
  - Primary/Secondary/Accent/Neutral만 사용
- 카드/패널:
  - 정보 밀도 높게: 섹션 타이틀 + 우측 액션(추가/내보내기/필터 저장)

  - 조합원/서비스/문의 상세에 “내부 메모(고객에게 보이지 않음)” 탭 제공
- 성능/가독성(디자인 레벨):

  - 문의 인박스 열기

(1) 조합원 리스트 + 관리(수정/삭제/블랙리스트 아웃처리)
[조합원 리스트 화면]
- 상단: 타이틀 “조합원 관리”
- 좌측: 필터 패널(접기/펼치기)
- 우측 상단: 액션 버튼
  - + 조합원 등록(수동 등록)
  - 내보내기(CSV/엑셀)
  - 뷰 저장
- 테이블 컬럼(기본):
  - 조합원ID
  - 이름
  - 생년(또는 연령대)
  - 연락처(마스킹: 010-****-1234, 권한 있으면 보기)
  - 주소(읍/면/동)
  - 상태(가입신청/입금확인중/가입완료/휴면/탈퇴/블랙리스트)
  - 가입일/신청일
  - 출자금(총액)
  - 최근 서비스 신청일
  - 메모 아이콘(내부)
  - 액션(보기/수정/상태변경)
- 필터(필수):
  - 상태
  - 가입일 범위
  - 지역(읍/면)
  - 출자금 입금 상태(미확인/확인)
  - 최근 활동(최근 30/90일)
- 벌크 액션(필수):
  - 상태 변경(휴면/활성)
  - 메모 추가
  - 내보내기
  - 블랙리스트 처리(사유 입력 필수)
  - 소프트 삭제(휴지통 이동)

[조합원 상세(우측 패널/상세 페이지)]
- 탭 구조:
  1) 개요
  2) 가입/상태 히스토리(타임라인)
  3) 출자금/입금 원장
  4) 서비스 신청/이력
  5) 문의/연락 이력
  6) 내부 메모
- 개요 카드:
  - 기본정보(이름/연락처/주소)
  - 상태 배지 + “상태 변경” 드롭다운
  - 빠른 버튼: 전화, 문자/카톡(연동은 UI만), 문의 생성, 서비스 신청 생성(대리 신청)
- 블랙리스트 처리 흐름:
  - 버튼 “블랙리스트로 전환”
  - 모달(2단계):
    1) 사유 선택(폭언/노쇼/부정/기타) + 상세 사유 입력
    2) 제한 범위 체크:
       - 서비스 신청 제한
       - 커뮤니티 참여 제한
       - 문의만 허용
    3) 적용일/해제일(옵션)
  - 적용 후: 고객용 화면에서 해당 제한이 표시된다고 가정(설명 텍스트 포함)

(2) 가입 승인 대기(운영OS 핵심)

  - 액션: 승인 / 보류(추가정보 요청) / 반려
- 승인 프로세스(디자인상):
  1) 신청 정보 확인
  2) 입금 매칭 여부 확인(바로 연결)
  3) 승인(가입완료로 상태 변경)
  4) 안내 발송(문자/카톡 템플릿 선택) UI 제공

- 테이블 컬럼:

  - 하단: 미니 네비(대시보드/서비스/문의/조합원)
- 모바일에서 가능한 주요 작업:
  1) 오늘 서비스 일정 확인(캘린더 간단)
  2) 새 문의 확인/템플릿 답변
  3) 조합원 검색 후 전화/메모 확인
  4) 입금 미매칭 확인(간단 목록)
- 모바일에서는 테이블을 카드 리스트로 변환(행 -> 카드).
- Operator Large Mode에서도 모바일 터치 영역(최소 48~56px) 준수.

10) 프로토타입 플로우(필수 시나리오 6개)
────────────────────────────────────────
1) 가입 승인:
   가입대기 리스트 → 신청자 상세 → 입금 원장 매칭 확인 → 승인 → 안내 발송 체크 → 상태 “가입완료”
2) 미매칭 입금 처리:
   입금 원장(미매칭 필터) → 수동 매칭 → 확인완료
3) 서비스 500건 처리:
   서비스 운영 콘솔 → ‘미배정’ 큐 → 벌크 선택 → 담당자 배정 → 시간 슬롯 일괄 확정 → 안내 발송
4) 서비스 캘린더 일정 변경:
   캘린더 이벤트 클릭 → 상세 패널 → 일정 변경 → 충돌 경고 해결 → 확정
5) 공지 작성 및 고객 미리보기:
   공지 작성(커버 이미지 필수) → 어르신 요약 입력 → 미리보기(일반/어르신) → 게시
6) 문의 답변:
   문의 인박스(새 문의) → 상세 → 템플릿 선택 → 답변 전송 → 상태 답변완료
